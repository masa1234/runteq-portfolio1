# 資格取得特化 学習習慣サービス 企画書

> 「Java Gold受験を申し込んだが、仕事が忙しくて勉強が止まっている人を、合格させるサービス」

---

## 1. サービス概要

| 項目 | 内容 |
|------|------|
| 概要 | 受験申込済みの社会人エンジニアが、試験日から逆算して「今日自分が合格ペースにいるか」を即座に確認できるWebサービス。記録よりも**ペース確認**が主目的。 |
| 誰の課題 | IPA・Oracle試験に申込済みで、受験料を払ったにもかかわらず学習が止まってしまっている社会人エンジニア |
| どう解決 | 「合格ペースか否か」をダッシュボード1画面で見せることで、毎日アプリを開くだけで不安と先送りを解消する |

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

Java Gold に申し込んだが、試験3週間前まで「まだ余裕ある」と思い込んでいた。
気づいたときには合格に必要な残り学習時間が物理的に足りず、受験をキャンセルした。

「今日1時間やれたかどうか」は分かる。
でも「それで合格ペースに乗れているのか」が全く分からないまま。
不安はあるのに、どれだけやれば安心なのかが見えなかった。

### 2-2. なぜそれが「気になった」のか

- **気づき：** 問題は意志力や記録の習慣ではなく、「今の自分が合格に間に合うのか」というフィードバックが存在しなかったこと
- **価値観：** 「やるべきことが見えていれば動ける」。構造がなかっただけだと気づいた
- **過去の経験：** Studyplus・Notion・手帳など複数の手段を試みたが、どれも「合格ペースか否か」を教えてくれなかった

---

## 3. 課題の整理

### 3-1. 表に見えている困りごと

資格の勉強が続かない・試験直前に焦って受験をキャンセルした

### 3-2. 本当に解決したい課題は何か

学習が止まる本当の原因（自身の経験から深掘り）

**❶「今日やらなくてもまだ大丈夫」という錯覚**
試験日まで時間があるように見えるため、毎日の先送りに罪悪感が生まれにくい。
「余裕感」が継続の最大の敵。

**❷「少しやった」が積み上がっている感覚がない**
1日30分やっても「焼け石に水では？」と感じる。
進捗が相対化されないため、小さな努力が報われた感がゼロ。

**❸ 忙しい日が続くと「もうダメだ」と諦める**
3日連続で学習できなかったとき、「もう間に合わない」という感覚で学習自体を諦める。
実際には間に合うかもしれないのに。

**→ 本当に解決したい課題：**
「今日サボっても大丈夫なのか、もう間に合わないのか」が分からない**不確実性による不安と諦め**。
今日の学習量ではなく、**合格ペースに乗れているかのフィードバック**が欠如している。

### 4-2. 最初に熱量高く使う人（ファーストユーザー）

**「受験料を払って申込済みだが、全然進んでいない人」**

- Java Goldなどを申込済みの社会人エンジニア
- 受験料（1.5〜2万円）を支払っており、失敗へのプレッシャーがある
- Udemyや参考書は買ったが、開けていない日が続いている
- 「あと○日しかない」と気づいて焦っている状態

**なぜこの人か：**
申込済みは「今すぐ使う理由」が存在する。
受験料という金銭的損失への恐怖が動機の後押しになる。
「広く使われるか」より「最初に刺さるか」を優先した。

### 4-3. 実際に使われる可能性について

- **最初に使ってくれそうな人：** 受験申込確認メールを見て「やばい、全然やってない」と気づいたエンジニア

- **思い出す場面②：** 「試験まであと○日」をカレンダーで確認して危機感を覚えた週明け
- **思い出す場面③：** Xで「Java Gold合格しました」の投稿を見た瞬間
- **今の代替手段：** Studyplus・Notion・手帳・何もしていない
- **乗り換える理由：** 「合格ペースか否か」を即座に判定してくれる手段が既存ツールに存在しない

### 4-4. ユーザーがサービスを導入して利用するまでのイメージ

## 7. このアプリで実現すること

### MVPの核となる価値

> **このアプリは学習記録アプリではない。**
> 「今日の自分が合格ペースにいるか」を教えてくれる**合格ペース管理ツール**。
> 記録は手段であり、目的はペース確認と不安解消。

---

### ⚠️ ペース判定の前提と限界

**① 判定はユーザーが設定した目標学習時間に対するものです**

ペース判定の基準となる「目標学習時間」はユーザー自身が登録する値です。
アプリ側で合格に必要な時間を定義しているわけではありません。
目標時間の設定にあたっては、各資格の公式情報・合格体験記・資格スクールの目安などを参考に、ご自身で適切な値を入力してください。

**② ペース判定は合格を保証するものではありません**

🟢「余裕」の判定は「設定した目標に対して順調なペースである」ことを示すものです。
学習の質・試験当日のコンディション・出題範囲の網羅度など、時間以外の要因は判定に含まれません。
あくまで「ペースの目安」としてご活用ください。

---

### MVPで最も検証したいこと

「学習記録が続くか」ではなく、**「ペース判定を見るために継続利用したくなるか」** を検証する。

ユーザーがアプリを開く動機が「記録するため」なのか「判定を確認するため」なのかを、リリース後の利用状況・フィードバックから確認する。
「判定を見たいから開く」という行動が確認できれば、このサービスの差別化が機能している証拠になる。

開発中も以下の問いを常に意識する：

- ペース判定バッジはダッシュボードで最初に目に入る位置にあるか
- 判定結果を見た瞬間に「今日どうすればいいか」が分かるか
- 記録の入力ステップは判定確認の邪魔になっていないか

---

### MVPで作る機能

- ユーザー登録 / ログイン
- 資格 / 試験日 / 目標時間の登録
- **ペース判定バッジ（🟢余裕 / 🟡要注意 / 🔴遅れ）← MVPの最重要機能**
- 今日のノルマ自動表示
- 学習記録の投稿（分数のみでOK）
- 全体進捗バー（累計 / 目標）

**ペース判定ロジック（Modelに集約）**

```ruby
# Certification モデル
def pace_status
  return :not_started if elapsed_days.zero?

  ideal_pace = target_minutes.to_f / total_days      # 1日あたりの理想学習時間
  actual_pace = total_studied_minutes.to_f / elapsed_days  # 実績ペース
  ratio = actual_pace / ideal_pace

  case ratio
  when 0.9..Float::INFINITY then :on_track    # 🟢 余裕
  when 0.6...0.9             then :caution    # 🟡 要注意
  else                            :behind     # 🔴 遅れ
  end
end

def daily_quota_minutes
  remaining = target_minutes - total_studied_minutes
  remaining_days = (exam_date - Date.current).to_i
  return 0 if remaining_days <= 0
  (remaining.to_f / remaining_days).ceil
end
```

> ※ 🔴「遅れ」判定時は「今日から毎日○分やれば間に合います」という再計算結果を必ずセットで表示する

### 本リリースで作る機能

- 週次ペースレポート
- 「このペースだと合格日が○日ずれる」予測表示
- 複数資格の同時管理
- リマインダー通知（メール）
- 合格報告 / SNSシェアカード

## 8. このアプリの懸念点とその対策

### 懸念点① 「遅れ」判定が逆にモチベを下げる

- **何が問題になりそうか：** 🔴が出ると「もう無理だ」と感じてアプリを閉じる可能性がある
- **なぜそう思うか：** 課題③「忙しい日が続くと諦める」と同じ心理が働く
- **対策：** 🔴判定でも「今日から毎日○分やれば間に合います」というポジティブな再計算結果を必ずセットで表示する。判定は現状報告であり、詰めるためのものではないと明示する。

### 懸念点② 試験後はアプリが不要になる（短命）

- **何が問題になりそうか：** 1つの試験に合格するとアプリを使わなくなる
- **なぜそう思うか：** 試験日という明確な期限があるサービスの宿命
- **対策：** 合格直後に「次の資格」を促すフローと、合格体験記（学習ログのサマリ）自動生成で「また使いたい」体験をつくる

## 10. 技術スタック

### 10-1. 使用予定の技術

| 項目 | 技術 |
|------|------|
| フレームワーク | Ruby on Rails 7.x |
| DB | PostgreSQL |
| デプロイ先 | Render / Railway（無料枠で始める） |
| 認証 | Devise gem |
| フロント | Hotwire（Turbo + Stimulus） |
| CSS | Tailwind CSS |

### 10-2. 技術選定の理由

- **Rails 7 + Hotwire：** SPA不要で、サーバーサイドレンダリングのままペース判定バッジの動的更新が実現できる
- **PostgreSQL：** 日付集計・期間クエリに強く、ペース計算の実装に適している
- **Tailwind CSS：** MVPの開発速度を優先できる

### 10-3. チャレンジしたい点と不安な点

- **チャレンジ：** ペース判定ロジック（`pace_status`）と再計算表示をModelに集約する設計
- **不安な点：** 日付をまたぐ際のノルマ再計算タイミング（日次バッチ vs リクエスト時計算）
  - → MVPでは「リクエスト時に毎回計算」でシンプルに実装し、パフォーマンス問題が出たら最適化する方針
