# 資格取得特化 学習習慣サービス 企画書

> 「忙しいエンジニアが、資格試験の合格から逆算して毎日の学習を記録・管理できるサービス」

---

## 1. サービス概要

| 項目 | 内容 |
|------|------|
| 概要 | 資格試験の合格日を設定し、残り日数から1日あたりの学習量を自動算出。毎日の学習を記録することで「今日やるべきことが明確」な状態を維持できるWebアプリ。 |
| 誰の課題 | 資格取得を目指しているが、業務の忙しさで学習が止まりがちな社会人エンジニア |
| どう解決 | 「合格から逆算した学習ペース管理」＋「記録の積み上げ可視化」で、漠然とした不安をなくし継続を支える |

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

AWSやIPA試験の勉強を始めたものの、「今日何をやればいいのか」が毎回不明確で、結局参考書を眺めて終わる日々が続いた経験。

「試験まであと2ヶ月あるからまだ大丈夫」という油断が続き、直前に焦って詰め込んで不合格になった悔しさ。

### 2-2. なぜそれが「気になった」のか

- **性格・価値観：** 「やるべきことが見えていれば動ける」という感覚。問題は意志力ではなく、構造の欠如だと気づいた
- **過去の経験：** 受験日は決まっているのに、毎日の学習量を自分で計算・管理するコストが高く、結果として先送りしてしまった

---

## 3. 課題の整理

### 3-1. 表に見えている困りごと

資格の勉強が続かない・試験直前に焦る

### 3-2. 本当に解決したい課題は何か

- **なぜ①：** 「今日何ページ進めればいいか」が毎回わからず、とりあえず開いて終わるから
- **なぜ②：** 試験日までの全体ボリュームと残り日数を自分で計算・管理するコストが高いから
- **なぜ③：** 進捗が見えないので「今日サボっても大丈夫」という錯覚が生まれ続けるから

**→ 本当に解決したい課題：**
「試験合格から逆算した、今日やるべき学習量が自動的に明示されること」。意志力ではなく構造の問題。

---

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

| 設定 | 内容 | 理由 |
|------|------|------|
| 年齢 | 25〜35歳の社会人エンジニア | AWS・基本情報・応用情報・TOEIC等の受験を検討または申込済みの層が最も切実 |
| 生活・状況 | 平日は業務で消耗。休日も予定が入りやすく計画的に進めにくい | 受験日は決まっているが、日々の業務との両立で学習が後回しになりやすい |
| 環境 | スマホ・PCブラウザ（アプリDL不要） | 通勤中に記録できる軽さが必須。インストールの摩擦をなくす |

### 4-2. 自分とユーザーの距離

**過去〜現在の自分。**
資格試験に申し込んでは消化不良で終わったことがある人すべてに刺さる課題であり、解像度が高い。

### 4-3. 実際に使われる可能性について

- **最初に使ってくれそうな人：** 受験申込直後の社会人エンジニア
- **思い出す場面：** 「試験まであと○日」と気づいた瞬間、または勉強できなかった日の夜
- **今の代替手段：** Studyplus・Notion・手帳・何もしていない
- **乗り換える理由：** 「今日のノルマが自動で出る」という体験は既存ツールにない。設計コストゼロで使い始められる

### 4-4. ユーザーがサービスを導入して利用するまでのイメージ

| フェーズ | 内容 |
|----------|------|
| きっかけ | 受験申込直後 or「試験まであと○日」と気づいた瞬間に検索・登録 |
| 最初の一歩 | 資格名・試験日・1日の学習可能時間を入力 → 今日のノルマが即表示される |
| 継続理由 | 「今日のノルマをこなしていない」という通知と、進捗バーが動く達成感の両方が効く |

## 5. 既存サービス・競合調査

### 5-1. 似たサービスの調査

| サービス名 | できること | 不満・ズレ |
|-----------|-----------|-----------|
| Studyplus | 学習時間の記録・グラフ表示 | 試験日からの逆算・今日のノルマ提示機能が弱い |
| Notion学習管理テンプレ | 自由な学習管理 | 自由度が高すぎて設計コストが高い。続かない |
| Udemy等の資格スクール | 学習コンテンツ提供 | 「進捗の個人管理」は手薄。自分でペースを作る必要がある |

### 5-2. それでも自分が作りたい理由

既存ツールはどれも「記録する場所」か「コンテンツを提供する場所」のどちらか。
「今日やるべき量を自動で決めてくれる」という体験が存在しない。
自分が本当に欲しかったのは、意志力ではなく「構造」だった。

### 5-3. 差別化を一文で決める

**「試験日と学習可能時間を入れるだけで、今日やるべきことが自動で決まる」という体験。コンテンツは持ち込み前提で、管理だけに特化する。**

## 7. このアプリで実現すること

### MVP で作る機能

- ユーザー登録 / ログイン
- 資格登録（資格名・試験日・目標学習時間）
- 今日のノルマ自動表示（残り日数から算出）
- 学習記録の投稿（分数・メモ）
- 全体進捗バー表示（累計 / 目標）

**今日のノルマ算出ロジック（MVPの核）**

```ruby
# Certification モデルに集約するイメージ
def daily_quota_minutes
  remaining_minutes = target_minutes - total_studied_minutes
  remaining_days = (exam_date - Date.current).to_i
  return 0 if remaining_days <= 0
  (remaining_minutes.to_f / remaining_days).ceil
end
```

> ※ 消化済みの記録を引いて毎日再計算。遅れた分は翌日以降に自動で再配分。

### 本リリースで作る機能

- 複数資格の同時管理
- 週次ペース分析（予定 vs 実績）
- リマインダー通知（メール）
- 合否記録・振り返りメモ
- SNSシェア（合格報告カード）

## 8. このアプリの懸念点とその対策

### 懸念点① 試験日が近づくと使われなくなる

- **何が問題になりそうか：** 直前期は焦りから記録より勉強優先になりがち
- **なぜそう思うか：** 記録という行動自体がオーバーヘッドに感じるタイミングが存在する
- **対策：** 記録を最速30秒（分数入力だけ）で完了できる設計にし、記録のコストをほぼゼロにする

### 懸念点② ノルマ未達が続くと離脱する

- **何が問題になりそうか：** ノルマを達成できない日が続くと罪悪感でアプリを開かなくなる
- **なぜそう思うか：** 記録系アプリの離脱の最大原因は「完璧にできない日が続くこと」
- **対策：** 未達でも翌日以降に自動再配分し「まだ間に合う」を常に表示。完璧主義にならない設計

## 10. 技術スタック

### 10-1. 使用予定の技術

| 項目 | 技術 |
|------|------|
| フレームワーク | Ruby on Rails 7.x |
| DB | PostgreSQL |
| デプロイ先 | Render / Railway（無料枠で始める） |
| 認証 | Devise gem |
| フロント | Hotwire（Turbo + Stimulus） |
| CSS | Tailwind CSS |

### 10-2. 技術選定の理由

- **Rails 7 + Hotwire：** SPA不要で、サーバーサイドレンダリングのまま動的なUI（進捗バー・ノルマ更新）が実現できる
- **PostgreSQL：** 日付集計・期間クエリに強く、ノルマ再計算の実装に適している
- **Tailwind CSS：** 素早くUIを組めるため、MVPの開発速度を優先できる

### 10-3. チャレンジしたい点と不安な点

- **チャレンジ：** 残り日数・ノルマの動的再計算ロジックをModelに集約する設計
- **不安な点：** 日付をまたぐ際のノルマ再計算タイミング（日次バッチ vs リクエスト時計算）
  - → MVPでは「リクエスト時に毎回計算」でシンプルに実装し、パフォーマンス問題が出たら最適化する方針
